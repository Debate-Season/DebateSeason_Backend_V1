## 0. 한눈에

**지금**: 이슈 하나에 채팅방(`chat_room`)이 여러 개(최대 3개) 매달려 있고, 방마다 채팅 내역이 갈라져 있다.
모바일은 이슈 안에서 방을 여러 개 본다.

**바꾼 뒤**: 이슈당 **컨테이너 채팅방 1개**만 남는다. 기존 방들은 **스레드**로 강등되고, 흩어져 있던
채팅이 컨테이너의 **하나의 전체 내역**으로 합쳐진다.

- **모바일**: 이슈당 채팅방 하나. 들어가면 전체 채팅 스트림(스레드 구분 없이).
- **웹**: 같은 채팅방에서 `[전체][주제A][주제B]` 탭으로 스레드별 필터 조회.
- **찬반·팀점수·MVP·알림·멤버십**: 스레드(기존 방)에 **그대로** 남는다 → 데이터 충돌·손실 없음.
- **메시지만** 컨테이너로 통합(`chat.chat_room_id → 컨테이너`, `chat.thread_id → 기존 방`).

**규모(2026-07-23 prod 실측)**: 이슈 17, 채팅방 26, **여러 방을 가진 이슈 8개**, 이슈당 최대 3방,
메시지 386건. → 마이그레이션은 순간. 관건은 데이터가 아니라 **실시간 주소·클라이언트 계약**(§6).

## 1. 목표 / 비목표

### 목표
1. 이슈당 컨테이너 채팅방 1개를 두고, 기존 다중 방을 그 아래 스레드로 강등한다.
2. 기존 채팅 내역을 컨테이너의 단일 전체 스트림으로 통합한다(메시지 유실 0).
3. 모바일: 이슈당 채팅방 하나 + 전체 스트림. 웹: 스레드 탭 필터.
4. 구 앱을 깨지 않는다(기존 방=스레드가 살아 있어 구 앱 투표·조회 경로 유지).

### 비목표 (이번 배포에서 안 함)
- **DebateTopic / DebateWiki / 리비전** 신규 엔티티 (v1.3.4 PRD §3.1·3.3) — §4 로 연기.
- **유저의 스레드(주제) 생성 기능** — 이번엔 기존 방을 스레드로 강등만. 신규 스레드 생성 API 없음.
- **찬반의 단위 변경** — 스레드(기존 방) 단위 그대로. `user_topic_opinion` 신설 안 함.
- Event(사건) / 미디어 클러스터링 / Presence — v1.3.4 PRD 소관.

## 2. 현황 (As-Is)

데이터·권한 현황 상세는 **v1.3.4 PRD §2** 를 그대로 따른다. 이 문서에 필요한 핵심만:

```
issue (1) ──< chat_room (N)   chat_room.issue_id       ← 채팅방 = 토론 주제 (한 엔티티가 안건=방=컨테이너 3역)
chat_room (1) ──< chat (N)    chat.chat_room_id        ← 메시지는 방에 소속
chat_room (1) ──< user_chat_room (N)                   ← 찬반(opinion)·관심(interest) = 방 단위
chat_room (1) ──< chatroom_notification_setting        ← 복합PK (user_id, chat_room_id)
```

- `chat_room_id` 하나가 **① STOMP 주소 ② 메시지 FK ③ 찬반·멤버십 키 ④ 알림 키** 를 동시에 짊어진다.
- `thread`/`topic`/`debate` 개념은 **코드에 아직 없다**(스키마·엔티티·API 전무).

## 3. To-Be 모델

### 3.1 컨테이너 + 스레드

```
issue ──< chat_room
            ├─ CONTAINER  (이슈당 1개, 신규 생성)   ← 모바일이 보는 "그 이슈의 채팅방", 메시지 버킷
            ├─ THREAD "전체" (이슈당 1개, 신규 생성) ← 모바일/무주제 글의 기본 착지
            └─ THREAD 주제A·B… (기존 chat_room 을 강등)  ← 투표·팀점수·MVP·멤버십·알림 그대로 보존
```

- **CONTAINER 는 새 `chat_room` 행**이다. 새 테이블을 만들지 않고 기존 방-기반 기계(STOMP·히스토리·FK)를
  그대로 재사용하기 위함. 컨테이너 자체는 투표를 갖지 않는다(메시지 버킷 + 이슈 진입점).
- **THREAD 는 기존 `chat_room` 행**을 그대로 둔 것이다. `type=THREAD` 로만 표시. 여기 걸린
  `user_chat_room`(찬반)·`chatroom_notification_setting`·집계 쿼리는 **손대지 않는다**.
- 방 구분용 컬럼을 `chat_room` 에 추가한다(§5.1). 조회 시:
  - 이슈의 컨테이너: `type=CONTAINER AND issue_id=?`
  - 이슈의 스레드 목록: `type=THREAD AND issue_id=?`

### 3.2 chat.thread_id (신규 컬럼)

```
ChatEntity + thread_id (nullable, FK → chat_room.chat_room_id = 스레드)
   chat.chat_room_id → CONTAINER   ← 마이그레이션 후 모든 메시지가 컨테이너 소속
   chat.thread_id    → THREAD      ← 원래 어느 방(주제)에서 온 메시지인지
```

- 마이그레이션 시 기존 메시지: `thread_id = 원래 chat_room_id`, `chat_room_id = 그 이슈의 컨테이너`.
- 모바일 신규 글: `thread_id = 전체 스레드`(기본). 웹 신규 글: `thread_id = 선택 탭`.

### 3.3 찬반·멤버십·알림 — 스레드에 그대로 (충돌 없음)

- `user_chat_room`(opinion AGREE/DISAGREE, interest), `chatroom_notification_setting`,
  팀점수/MVP/하이라이트 집계(`ChatRoomRepository`) — **전부 스레드(기존 방) 키 그대로 유지**.
  → N→1 붕괴에 따른 dedup/PK 충돌이 **아예 발생하지 않는다.**
- 컨테이너의 "멤버십"은 저장하지 않고 **조회 시 파생**한다: 이슈의 어느 스레드든 가입/투표했으면
  컨테이너 참여자. 알림도 스레드 설정의 **합집합**(하나라도 ON → 이슈 알림 ON).

### 3.4 실시간 (STOMP) — 목적지 패턴 유지, payload 에 threadId 추가

현재: 발행 `@MessageMapping("/chat.room.{roomId}")`, 구독 `@SendTo("/topic/room{roomId}")`
(`WebSocketControllerV1.java:37-38`). `{roomId}` 가 곧 `chat_room_id`.

**변경 최소안(권장)**:
- 브로드캐스트는 **컨테이너 주소 하나** `/topic/room{containerId}` 로 유지.
- **응답 payload 에 `threadId` 필드만 추가**한다.
- 모바일: 컨테이너 구독 → 전부 표시. 웹: 같은 컨테이너 구독 → `threadId` 로 탭 필터(클라이언트 사이드).
- → STOMP 목적지 문법을 안 바꾼다. `@SendTo` 값도 그대로. **재구독/주소 체계 변경 없음.**

> 대안(스레드별 목적지 `/topic/room{c}.thread{t}`)은 구독 granularity 를 서버가 갖지만
> 컨트롤러·클라이언트 변경이 커진다. 웹 탭 전환이 클라이언트 필터로 충분하므로 **최소안 채택.**

### 3.5 조회 모드

| 모드 | 쿼리 | 용도 |
|---|---|---|
| 전체 스트림 | `WHERE chat_room_id = :containerId` | 모바일 기본 화면 / 웹 [전체] 탭 |
| 스레드 필터 | `WHERE chat_room_id = :containerId AND thread_id = :threadId` | 웹 주제 탭 |

- 전체 스트림은 **기존 커서 쿼리 구조 그대로**(`findByRoomIdAndCursor`) — roomId 에 컨테이너 id 를 넣으면 됨.
- 스레드 필터는 `thread_id` 조건이 붙은 쿼리를 신설. **커서는 모드별 독립**(전체 커서를 스레드 뷰에 재사용 금지).
- **인덱스**: `chat(chat_room_id, chat_id)`(전체 스트림 커서) + `chat(thread_id, chat_id)`(스레드 필터) 추가.
  `chat` 은 가장 빨리 커지는 테이블 — 인덱스 없이 열면 즉시 느려진다(v1.3.4 PRD §6.5 지적).
- **날짜 조회 주의**: `findByRoomIdAndCursorAndDate` 가 `DATE(time_stamp)=` 로 인덱스를 못 탄다.
  스레드 필터에 같은 패턴 복사 금지 — 범위 조건(`time_stamp >= :start AND < :end`)으로.

### 3.6 API 계약 변화

| 엔드포인트 | 변화 | 호환 |
|---|---|---|
| 채팅 히스토리 `GET /api/v1/chat/rooms/{roomId}/messages` | roomId=컨테이너 시 전체 스트림. `?threadId=` 옵션 추가로 스레드 필터. 응답 메시지에 `threadId` 필드 추가 | 응답 필드 추가 = 안전. 구 앱은 무시 |
| STOMP 브로드캐스트 payload | `threadId` 필드 추가 | 안전 |
| 메시지 전송 `ChatMessageRequest` | `threadId`(optional) 추가 | 구 앱 미전송 → 기본 스레드 착지 |
| 방 목록/입장 (`findVotedChatRoom`, `GET /api/v1/room`) | **신규 클라이언트에 컨테이너를 노출**해야 함 | §6 — 신 경로/헤더로 분기 |

## 4. v1.3.4 PRD 와의 관계 (반드시 읽을 것)

v1.3.4 PRD(§3.1·3.2·3.7)는 스레드를 **DebateTopic 신규 엔티티**로, 찬반을 **`user_topic_opinion`** 으로
설계했다. v1.3.5 는 그 그림의 **경량 선행 단계**로, 다음을 **의도적으로 다르게** 한다:

| 항목 | v1.3.4 PRD (전체 그림) | v1.3.5 (이번) |
|---|---|---|
| 스레드 실체 | 신규 `debate_topic` 테이블, 유저 생성 | **기존 `chat_room` 을 강등** (신규 테이블 없음) |
| chat 의 스레드 FK | `topic_id → debate_topic` | `thread_id → chat_room`(스레드) |
| 찬반 | `user_topic_opinion` 로 이관 | **`user_chat_room` 그대로**(스레드=방이므로 이미 주제 단위) |
| 위키/리비전 | 신설 | 안 함 |
| 유저 스레드 생성 | 있음 | 없음(강등만) |

**왜 경량으로?** 지금 필요한 것은 "흩어진 방을 하나로 합쳐 모바일에 단일 채팅방을 주는 것"뿐이다.
기존 방들은 이미 title/content/created_by/status/투표를 가진 **사실상의 경량 주제**이므로,
새 테이블 없이 그대로 스레드로 쓰는 것이 최소 변경이다.

**미래 경로 (DebateTopic 로 승격 시)**: 강등된 방(스레드)들이 곧 seed DebateTopic 이 된다.
그때 `thread_id`(→chat_room) 를 `topic_id`(→debate_topic)로 옮기거나, `debate_topic` 를
강등된 방에 1:1 로 얹으면 된다. **`thread_id` 컬럼명을 특정 테이블에 종속되지 않게 유지**하면 전환 비용이 준다.

> **결정 포인트(§10-1)**: 이 divergence 를 감수할지, 아니면 처음부터 `debate_topic` 을 만들지.
> 데이터 규모(스레드 26건)가 작아 나중 승격도 저비용이므로 **경량 선행을 권장.**

## 5. 마이그레이션 (Expand / Migrate / Contract)

원칙은 v1.3.4 PRD §6.2 를 그대로 따른다. **`chat_room` PK 는 절대 건드리지 않는다.**

### 5.1 Expand — 스키마 추가 (하위호환, 무해)

```sql
-- chat_room: 컨테이너/스레드 구분 + 컨테이너 역참조
ALTER TABLE chat_room ADD COLUMN room_type VARCHAR(16) NULL;      -- 'CONTAINER' | 'THREAD' | NULL(레거시=THREAD 취급)
ALTER TABLE chat_room ADD COLUMN container_room_id BIGINT NULL;   -- THREAD → 소속 컨테이너 (CONTAINER 는 NULL)

-- chat: 스레드 참조 + 커서 인덱스
ALTER TABLE chat ADD COLUMN thread_id BIGINT NULL;               -- FK → chat_room(스레드). NULL=미분류
ALTER TABLE chat ADD INDEX idx_chat_room_cursor (chat_room_id, chat_id);
ALTER TABLE chat ADD INDEX idx_chat_thread_cursor (thread_id, chat_id);
```

- 386행 / MariaDB 10.6 → `ADD COLUMN`·`ADD INDEX` 모두 순간(v1.3.4 PRD §6.5).
- 이 단계는 구/신 클라이언트 **모두 무영향**. 코드는 `thread_id` 있으면 저장, 없으면 기존대로.

### 5.2 Migrate — 데이터 이관 (프로덕션 쓰기 — **별도 승인 필수**)

이슈별로(여러 방을 가진 8개 이슈가 핵심, 단일 방 이슈도 균일 처리):

```
FOR EACH issue:
  1. 컨테이너 생성: chat_room(room_type=CONTAINER, issue_id, title=이슈제목, created_by=NULL)
  2. "전체" 스레드 생성: chat_room(room_type=THREAD, issue_id, container_room_id=컨테이너, title='전체')
  3. 기존 방들 강등: UPDATE chat_room SET room_type='THREAD', container_room_id=컨테이너
                     WHERE issue_id=? AND room_type IS NULL
  4. 메시지 재지정: UPDATE chat SET thread_id = chat_room_id, chat_room_id = 컨테이너
                   WHERE chat_room_id IN (그 이슈의 강등된 방들)
```

- **백업 먼저**: `chat`, `chat_room` 스냅샷(mysqldump 또는 `CREATE TABLE _bak AS SELECT`).
- **dry-run**: 트랜잭션 안에서 실행 후 건수 대조 → 문제없으면 커밋. 롤백 스크립트 동반.
- **검증**: 이관 전/후 `COUNT(chat)` 동일, 이슈별 메시지 합 보존, 컨테이너당 스레드 수 = 강등방+1(전체).
- **찬반/멤버십/알림은 건드리지 않는다** — 스레드에 그대로 남으므로 이관 대상 아님(충돌 원천 제거).

### 5.3 Contract — 레거시 제거 (스토어 반영 + force_update 이후)

- 신규 클라이언트 확산 확인 후, 구 방-목록 경로/구 조회 분기 정리.
- 자세한 시점 규칙은 v1.3.4 PRD §6.1-1(배포 시차) 그대로.

## 6. 클라이언트 호환성

v1.3.4 PRD §6.1~§6.4 의 프레임(Expand/Migrate/Contract, 강제 업데이트의 한계, 실측)을 그대로 적용한다.
**v1.3.5 는 "방 재활용 + 스레드별 투표 유지" 덕분에 구 앱 파손 위험이 v1.3.4 안보다 낮다:**

- 구 앱이 실제로 쓰는 쓰기 API 는 `POST /api/v1/room/vote` 하나(§6.3-1 실측). 이 방들이 스레드로
  **그대로 살아 있으므로** 구 앱 투표·`GET /api/v1/room?chatroom-id=<옛방>` 조회는 **무변경으로 동작**한다.
- 즉 v1.3.4 PRD §6.4 의 "대표 주제 shim" 이 **거의 필요 없다** — 구 앱은 자기가 알던 방(=스레드)을 계속 쓴다.

남는 유일한 호환 과제: **방 목록/진입에서 신규 클라이언트에 컨테이너를 노출**하는 것.
- 구 앱: 지금처럼 스레드(옛 방)들이 목록에 뜨거나(무변경), 또는 그대로 유지.
- 신 앱/웹: 컨테이너 1개를 본다.
- 분기 수단: `/api/v2/room` 신규 엔드포인트(레포에 `/api/v2/users/login` 전례) **또는**
  `X-Client-Version` 헤더(v1.3.4 PRD §6.1-2). → **§10-2 결정**.

## 10. 열린 결정

**10-1. 스레드 실체 — 경량(기존 방 강등) vs 정식(DebateTopic 신설)**
- 권장: **경량**. 스레드 26건 규모라 나중 DebateTopic 승격도 저비용. 이번엔 통합만.

**10-2. 신규 클라이언트에 컨테이너를 어떻게 노출하나**
- (a) `/api/v2/room` 신규 엔드포인트 (레포 전례 있음) — 구/신 완전 분리, 명확
- (b) `X-Client-Version` 헤더 분기 — 엔드포인트 안 늘리나 서버가 버전 인지 필요
- 권장: **(a)** — 계약이 명시적이고 Contract 시 구 엔드포인트만 접으면 됨.

**10-3. "전체" 스레드의 의미**
- 모바일/무주제 글이 착지하는 기본 스레드. 웹 [전체] 탭 = 이 스레드만 볼지, **모든 스레드 합본**을 볼지.
- 권장: 웹 [전체] = 컨테이너 전체 스트림(모든 스레드 합본), 주제 탭 = 해당 thread_id 필터.
  모바일 기본 글의 thread_id 는 "전체" 스레드 id.
