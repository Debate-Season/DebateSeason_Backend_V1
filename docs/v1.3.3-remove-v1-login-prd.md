## 1. 문제

`POST /api/v1/users/login` 은 요청 본문의 소셜 ID 문자열만 받아 **아무 검증 없이 토큰을 발급**했다.

```json
{ "identifier": "1323412", "socialType": "kakao" }
```

`UserServiceV1.socialLogin` 의 흐름:

```java
User user = userRepository.findByIdentifier(request.identifier());  // 그냥 DB 조회
if (user == User.EMPTY) {
    user = User.create(request.identifier(), request.socialType()); // 없으면 생성
} else {
    user.login();
}
String accessToken = jwtUtil.createAccessToken(user.getId(), user.getRole());  // 토큰 발급
```

**"이 요청자가 그 소셜 계정의 주인인가"를 확인하는 단계가 없다.**
카카오/애플에 확인하지 않고, 서명 검증도 없다. 문자열 하나가 곧 로그인이었다.

### 1.1 가능했던 공격

| | 내용 |
|---|---|
| **계정 탈취** | 남의 소셜 ID 를 알면 그 사람 토큰을 받는다. 프로필 조회, 그 이름으로 채팅·투표, `POST /api/v1/users/withdraw` 로 **탈퇴까지** 가능 |
| **무제한 계정 생성** | 임의 문자열이 그대로 신규 계정이 된다 |
| **가입 여부 확인** | 응답의 `profileStatus`/`termsStatus` 로 특정 소셜 ID 가 기존 가입자인지 식별 가능 |

### 1.2 근본 원인

**소셜 ID 는 신원을 "식별"하는 값이지 본인임을 "증명"하는 값이 아니다.**
사용자가 카카오 로그인을 쓴 다른 서비스, 클라이언트 네트워크 트래픽 등 여러 곳에 남는다.
비밀번호처럼 다뤄서는 안 되는 값을 인증 수단으로 사용한 것이 원인이다.

### 1.3 노출 범위 (실측)

`users` 테이블 집계 — 값은 조회하지 않고 형식만 확인했다.

| 소셜 | 인원 | identifier 길이 | 순수 숫자 |
|---|---|---|---|
| KAKAO | 128 | 7~11자 | 120 / 128 |
| APPLE | 103 | 8~44자 | 0 |

- **애플**은 `sub` 가 길고 불규칙해 추측이 사실상 불가능하다.
- **카카오**는 7~11자리 숫자이고 순차 발급이라, **하나를 알면 인접 값 탐색이 유효**하다.
- 익명 요청은 분당 100회로 제한되지만(`RateLimitFilter`, IP 기준) IP 를 바꾸면 우회된다.
  애초에 소셜 ID 는 비밀이 아니므로 전수 탐색이 필요하지도 않다.

### 1.4 왜 지금인가

현재 모든 사용자가 `USER` 라 이 경로로 관리자 권한을 얻을 수는 없다.
그러나 **v1.3.4 에서 인가 매처를 켜려면 ADMIN 계정을 만들어야 하고, 그 순간
"ADMIN 의 소셜 ID 를 알면 관리자로 로그인된다"가 되어 심각도가 올라간다.**
따라서 ADMIN 승격보다 먼저 막아야 한다.

## 2. V2 와의 차이 — 딱 한 줄

```java
// V1 (제거 대상): 클라이언트가 준 값을 그대로 신뢰
String identifier = request.identifier();

// V2 (정상): 서명된 id_token 에서 서버가 직접 추출
String identifier = oidcProviderFactory.extractUserId(request.socialType(), request.idToken());
```

이 한 줄 아래로는 두 서비스의 코드가 동일하다.
id_token 은 카카오/애플이 서명하므로 위조하면 검증에서 걸린다.

## 3. 사용처 확인

**V1 로그인을 호출하는 클라이언트가 없다.**

| 클라이언트 | 로그인 경로 |
|---|---|
| `DebateSeason_Frontend_V1` (Flutter) | `POST /api/v2/users/login` |
| `toronchul_web_v1` (Next.js) | `POST /api/v2/users/login` |
| `docs/web-login-api.md` | "OIDC 기반의 V2 API를 사용합니다" |

서버 로그 7일치에도 V1 로그인 호출이 잡히지 않았다.
다만 **Spring 이 모든 요청을 로깅하지는 않으므로 "사용 0건"의 증거로는 불완전하다.**
이 불확실성이 아래 설계 판단의 근거다.

## 4. 변경 내용

### 4.1 검증 없는 로그인 로직 삭제

- `UserServiceV1.socialLogin` **메서드 자체를 삭제**했다.
  컨트롤러만 막고 서비스 로직을 남기면 다른 경로에서 되살아날 수 있다.
- 전용 DTO 2개(`SocialLoginRequest`, `SocialLoginServiceRequest`)도 함께 삭제했다.

### 4.2 404 가 아니라 410 Gone

엔드포인트를 통째로 지우는 대신, 호출 시 `ErrorCode.REMOVED_API`(410 GONE)를 던진다.

```
지원이 종료된 API입니다. 앱을 최신 버전으로 업데이트해주세요.
```

**이유**: §3 에서 보듯 로그만으로는 사용 0건을 증명할 수 없다.
만에 하나 남아 있을 구버전 클라이언트에게 조용한 404 대신 **무엇을 해야 하는지 알려주는 편**이 낫다.
동시에 취약한 로직은 이미 삭제됐으므로 **구멍은 즉시 닫힌다.**

`/api/v1/users/login` 은 `PUBLIC_URLS` 에 그대로 둔다.
빼면 토큰 없는 요청이 401 이 되어 410 메시지가 전달되지 않는다.

### 4.3 호출 감지

410 은 4xx 이므로 v1.3.2 에서 만든 규칙에 따라 `GlobalExceptionHandler` 가 **WARN** 으로 남긴다.

```
CustomException. code=REMOVED_API status=410 uri=/prod/api/v1/users/login message=지원이 종료된 API입니다...
```

Discord 알림은 가지 않는다(클라이언트 오류이므로 의도한 동작).
대신 배포 며칠 뒤 아래로 실사용 여부를 확인한다.

```bash
ssh debateseason-prod "sudo journalctl -u toronchul.service --since '7 days ago' --no-pager | grep -c REMOVED_API"
```

**0 이면** 엔드포인트와 `REMOVED_API` 코드를 완전히 삭제한다(후속 정리).
**0 이 아니면** 어떤 클라이언트인지 파악한 뒤 대응한다(`X-Client-Version` 로깅이 v1.3.1 에 들어가 있다).

### 4.4 테스트

`RemovedV1LoginTest`

| 케이스 | 검증 |
|---|---|
| 소셜 ID 로 로그인 시도 | 410, `code=REMOVED_API`, **응답에 토큰 없음**, 서비스 미호출(`verifyNoInteractions`) |
| 본문 없이 호출 | 410 |
| `UserServiceV1.socialLogin` 존재 여부 | **메서드가 코드에 없어야 함** — 리플렉션으로 확인 |

마지막 케이스는 "컨트롤러만 막고 로직은 남기는" 회귀를 잡기 위한 것이다.

## 6. 배포

`main` push → PROD_CI/CD 자동 배포. 마이그레이션 없음.

### 검증

```bash
# 410 과 안내 메시지가 나와야 한다
curl -s -X POST "https://api.toronchul.app/prod/api/v1/users/login" \
  -H 'Content-Type: application/json' \
  -d '{"identifier":"1323412","socialType":"KAKAO"}'

# V2 는 정상 동작해야 한다 (id_token 이 없으므로 401/400 이 나오면 정상 — 500 이 아니어야 한다)
curl -s -o /dev/null -w "%{http_code}\n" -X POST "https://api.toronchul.app/prod/api/v2/users/login" \
  -H 'Content-Type: application/json' -d '{}'
```

**배포 후 실제 로그인 1회**로 V2 경로가 멀쩡한지 확인할 것.
(v1.3.1 의 role 클레임 검증도 아직 남아 있어 겸사겸사 확인된다)

### 롤백

애플리케이션만 되돌리면 된다. 다만 되돌리면 인증 우회가 다시 열리므로,
문제가 생기면 롤백보다 **V2 경로를 고치는 방향**이 옳다.
