### 0.3 블로킹 결정 3건 — §7 2번(DebateTopic) 착수 전 필요

**결정 1. 토론 주제를 누가 닫는가?**

유저가 주제를 만들면 위키가 "진행 중"으로 생기고, 닫히면 아카이브된다. 닫는 주체는?

| 후보 | 문제 |
|---|---|
| 작성자 본인 | 주제는 만든 순간 **공공재**다. 남들이 투표·발언했는데 작성자가 닫으면 그 참여를 일방적으로 끝낸다. 안 닫고 떠나면 영원히 OPEN |
| ADMIN | 품질은 좋으나 운영 부담. 어드민 UI 도 없음 |
| 무활동 자동 종료 | 마지막 활동 후 N일(예: 7일)이면 자동 CLOSED. 방치 문제 해결 |

**추천: 자동 종료 + ADMIN 수동. 작성자에게는 닫을 권한을 주지 않는다.**
자동 종료를 택하면 `debate_topic.last_activity_at` 컬럼과 스케줄러가 추가로 필요하다.
(탈퇴 익명화가 매일 자정에 도는 배치 자리가 이미 있다)

**결정 2. 주제가 CLOSED 된 뒤 그 스레드에 계속 발언할 수 있는가?**

| 후보 | 문제 |
|---|---|
| 읽기 전용 | 명확하다. 더 얘기하려면 새 주제를 열면 된다(유저가 주제를 만들 수 있어 비용이 낮다) |
| 발언 계속 + 투표만 마감 | **위키는 아카이브인데 스레드는 살아 있는 모순.** 아카이브된 본문과 이후 대화가 어긋나기 시작한다 |

**추천: 읽기 전용.** 채팅 전송 경로(HTTP·WebSocket 양쪽)에서 주제 상태 확인이 필요해진다.

**결정 3. 대표 주제가 CLOSED 되면 구 앱 투표는 어디로 가는가?**

> ⚠ **이 항목은 추천이 바뀌었다.** 처음에는 (a) 자동 이동을 권했으나, 결정 1(자동 종료)과
> 함께 놓고 보면 위험하다. 유저는 "이 채팅방 안건"에 투표한 줄 아는데 표가 **전혀 다른 주제로**
> 들어간다. 데이터 오염이다.

| 후보 | 평가 |
|---|---|
| (a) 최근 OPEN 주제로 자동 이동 | **비권장.** 위 사유 |
| (b) 투표 거부 | 정직하나 구 앱 UX 가 깨진다 |
| (c) **대표 주제는 자동 종료 대상에서 제외** | **권장.** 채팅방이 살아 있는 한 대표 주제도 OPEN 유지 |

**추천: (c).** "대표 주제 = 이 채팅방의 원래 안건"이므로 의미상으로도 정확한 매핑이다.
구 앱이 완전히 사라진 뒤(강제 업데이트 후) Contract 단계에서 정리한다.

## 1. v1.3.4가 노리는 것

이슈(Issue)를 **허브**로 승격시키고, 그 아래에 **사건 / 채팅방 / 미디어**를 붙인다.
그리고 채팅방 안에서 **유저가 토론 주제를 만들면 토론위키 페이지가 한 장 생성**되고, 주제가 닫히면 그 페이지가 아카이브되는 흐름을 만든다.

현재는 `Issue → ChatRoom → Chat` 일직선이고, `ChatRoom` 엔티티 하나가 "안건 = 토론방 = 채팅 컨테이너" 3역을 겸한다.
모바일의 **토론방 화면이 곧 이 `ChatRoom` 상세**이며, v1.3.4에서 이 화면이 **토론위키**가 된다.

### 목표 구조 (To-Be)

```
Issue (허브)
 ├─ Event(사건)         [신규]   이슈의 타임라인을 구성하는 하위 단위
 ├─ Media              [신규 연결] 이슈로 클러스터링된 뉴스/유튜브/커뮤니티
 └─ ChatRoom(채팅방)     [1:N, ADMIN만 생성]  ← 유저 생성 불가
      ├─ Chat(일반 발언)          topic_id = NULL
      └─ DebateTopic(토론 주제)  [신규, 유저가 생성, 0개일 수 있음]
           ├─ Chat(스레드 발언)   topic_id = N
           ├─ DebateWiki(토론위키 페이지)  주제 1건 = 위키 1장
           │    └─ OPEN → ACTIVE(진행 중) / CLOSED → ARCHIVED(아카이브)
           └─ UserTopicOpinion(찬반)      ← 찬반은 주제 단위
```

핵심 규칙:
- **채팅방 생성은 ADMIN 전용.** 유저가 만드는 것은 **토론 주제**뿐이다.
- **토론 주제는 스레드다.** 주제에 달린 발언은 전체 채팅방 스트림에서도 보이고, 해당 스레드로 필터링해서도 볼 수 있다.
- **토론 주제는 있을 수도, 없을 수도 있다.** 주제 0개인 채팅방은 정상 상태이며, 그때는 평범한 채팅방으로 동작한다.
- **찬반은 토론 주제 단위.** 한 유저가 같은 채팅방 안에서 주제A는 찬성, 주제B는 반대일 수 있다.
- **토론위키는 개념은 하나, 페이지는 여러 장.** 한 이슈에서 발생한 토론 주제 수만큼 위키 페이지가 쌓인다.
- **채팅 참여에 찬반 투표는 필요 없다.**
- 채팅방 진입 시 소속 **이슈 정보 + 해당 채팅방의 위키(진행 중/아카이브) 목록**을 함께 로드한다.

## 2. 현황 (As-Is)

### 2.1 데이터 구조

```
IssueEntity (issue)
 ├ issue_id, title, major_category, middle_category, created_at
 └ 작성자 없음 / 상태 없음 / updated_at 없음

ChatRoom (chat_room)             ── ManyToOne ─> IssueEntity
 ├ chat_room_id, title, content, created_at
 └ "안건 = 토론방 = 채팅방" 이 한 엔티티에 혼재. 작성자·상태 없음

ChatEntity                       ── ChatRoom, user_id, opinion_type, user_community
UserIssue                        ── User × Issue,     bookmark : String
UserChatRoom                     ── User × ChatRoom,  opinion : String, interest : String
MediaEntity (media)              ── 이슈와 연결 없음. category(String) 만 보유
ReportEntity                     ── ReportStatus(NONE/PENDING/PROCESSING/ACCEPTED/REJECTED)
```

관련 enum: `Category = 정치,사회,경제,세계` / `Opinion = AGREE,DISAGREE,NEUTRAL`
`middleCategory` 는 엔티티에만 있고 `IssueRequest` 에서는 주석 처리 (미사용).

### 2.2 "토론방"의 현재 정체

별도 엔티티가 **없다.** 모바일 토론방 화면은 `GET /api/v1/room?chatroom-id=` 하나가 그리고 있고,
응답 `ChatRoomResponse` 가 `title / content / agree / disagree / teams / opinion(내 투표)` 를 담는다.
`ChatRoomControllerV1.java:37` 주석에 "채팅방(=안건=토론방)생성하기" 라고 명시돼 있다.

그 화면에 붙을 예정이던 위키 탭은 **주석 처리된 채 미구현**이다:
- `ChatRoomControllerV1.java:62` — `// type은 토론위키일 수도 있고, 하이라이트일 수도 있고, 아무것도 없을 수도 있다.`
- `ChatRoomControllerV1Docs.java:61` — `description = "토론위키(wiki)냐 하이라이트(highlight)냐 없어도 상관없음"` (블록 주석 안)
- `ChatRoomResponse` 의 `private HightlightResponse highlight;` 도 주석 처리

즉 v1.3.4의 위키화는 **새 화면을 만드는 게 아니라, 지금의 `ChatRoom` 을 채팅 컨테이너와 위키 페이지로 쪼개는 작업**이다.

### 2.5 확인된 결함

**v1.3.1 에서 해결됨**
- ~~작성자 미기록~~ → `created_by` / `status` / `updated_at` 컬럼 추가, 생성 시 principal 기록
- ~~신고자 위조 가능~~ → `@AuthenticationPrincipal` 로 교체 + 회귀 테스트
- ~~`@AuthenticationPrincipal Long userId` 오용~~ → `CustomUserDetails` 로 교체, 토큰 재파싱 fallback 제거
- ~~레거시 STOMP 스푸핑 경로~~ → 핸들러 제거 (클라이언트 사용처 0건 확인)
- ~~역할(Role) 부재~~ → `UserRole(USER/ADMIN)` + JWT role 클레임

**v1.3.4 에서 처리할 것**
1. **인가 부재** — `POST /api/v1/issue`, `POST /api/v1/room` 이 여전히 토큰만 있으면 누구나 호출 가능.
   v1.3.1 에서 role 은 만들었으나 **인가 매처를 아직 걸지 않았다.**
2. **수정/삭제 API 부재** — `created_by` 가 생겼으므로 이제 소유권 판정이 가능하다.
3. **`status` 조회 필터 미적용** — 컬럼만 있고 목록/조회 쿼리는 아직 필터하지 않는다.
4. **타입 안전성** — `opinion`/`interest`/`bookmark` 가 String, `@Enumerated` 미적용. `(user, chat_room)` UNIQUE 인덱스 부재 가능성 (`chat_reaction` 유사 전례 → `docs/chat-reaction-unique-index-issue.md`).
5. **엔티티 캡슐화** — `ChatRoom`, `UserChatRoom`, `UserIssue`, `MediaEntity` 모두 `@Setter` + `@AllArgsConstructor` 전면 공개.

> WebSocket 인증은 상대적으로 양호. prod 는 `chat.websocket.auth-required: true` 로 CONNECT 시 JWT 검증하고,
> `ChatServiceV1.processChatMessage()` 가 sender 를 서버 프로필에서 채운다.

## 3. 구조 설계

### 3.1 DebateTopic(토론 주제) — v1.3.4의 중심

유저가 만드는 유일한 토론 단위. 채팅방 안에서 발생한다.

```
DebateTopic (debate_topic)
 ├ topic_id
 ├ chat_room_id        ← 채팅방에 부착
 ├ issue_id            ← 비정규화 (이슈 단위 위키 목록 조회용, 3.3 참조)
 ├ title               토론 주제 (= 위키 페이지 제목)
 ├ status              OPEN / CLOSED
 ├ opened_at, closed_at
 ├ agree_count, disagree_count   ← 비정규화 카운터 (선택, 3.2 참조)
 ├ created_by          ← 유저
 └ created_at, updated_at
```

**생명주기**

```
유저가 토론 주제 생성
   → DebateTopic(status=OPEN) 생성
   → DebateWiki 페이지 자동 생성 (status=ACTIVE, "진행 중")
   → 그 시점부터 해당 주제에 찬반 투표 가능

토론 주제 종료
   → DebateTopic.status = CLOSED
   → DebateWiki.status = ARCHIVED
   → 찬반 투표 마감 (기존 표는 보존, 결과는 계속 조회 가능)
```

**주제 개수는 0개 이상 N개.** 스레드 모델이므로 동시에 OPEN 인 주제가 여러 개 있어도 무방하고,
주제가 하나도 없는 채팅방은 **평범한 채팅방으로 동작**한다(모든 메시지가 `topic_id = NULL`).
따라서 모든 조회 경로는 **주제 0건을 정상 상태로 취급**해야 한다 — 채팅방 상세의 `topics: []`, 위키 목록 빈 배열이 에러가 아니다.

> **결정 필요**: 주제를 **누가/어떻게 닫는가?** 후보 — (a) 작성자 본인, (b) ADMIN, (c) 일정 시간 무활동 시 자동, (d) 조합.
> 자동 종료를 넣으려면 스케줄러가 필요하고, `domain/youtubeLive/scheduler` 처럼 기존 배치 위치에 붙일 수 있다.

> **결정 필요**: 주제가 CLOSED 된 뒤 그 스레드에 **댓글을 계속 달 수 있는가?**
> 찬반 투표는 마감하되 대화는 열어두는 선택지도 있다. 위키가 ARCHIVED 된 뒤에도 스레드가 살아 있으면
> "아카이브"의 의미가 흐려지므로, **스레드도 함께 읽기 전용 전환**을 권장.

### 3.2 찬반 — 주제 단위로 이동

```
UserTopicOpinion (user_topic_opinion)
 ├ id
 ├ user_id
 ├ topic_id
 ├ opinion          @Enumerated(STRING)  AGREE / DISAGREE
 └ created_at, updated_at
 UNIQUE (user_id, topic_id)     ← 반드시 걸 것
```

- `Opinion` enum 의 `NEUTRAL` 은 **저장하지 않는다.** "행이 없음 = 미투표 = NEUTRAL" 로 해석 (현재 `ChatRoomServiceV1.fetch()` 도 이미 그렇게 동작).
- 집계는 `COUNT` 쿼리로 시작하고, 채팅방 상세에서 주제 목록을 매번 집계하면 N+1 이 되므로 **`debate_topic` 에 카운터를 비정규화**하는 것을 권장. 투표 시 카운터를 함께 갱신.
- **기존 `UserChatRoom.opinion` 데이터 이관**: 각 채팅방의 title/content 를 그대로 "첫 토론 주제"로 만들고(3.5 참조), 기존 표를 그 주제로 옮긴다. `UserChatRoom` 은 `interest`(관심등록)만 남긴다.

### 3.3 DebateWiki(토론위키) — 주제당 한 장

위키와 주제는 **1:1**이다. 그렇다면 `debate_topic` 에 본문 컬럼을 붙이면 그만 아닌가? — **아니다.**
위키는 **개정 이력**이 본질이므로, 메타(주제·상태·찬반)와 본문(리비전)을 나눈다.

```
DebateWiki (debate_wiki)
 ├ wiki_id
 ├ topic_id            ← 1:1
 ├ status              ACTIVE(진행 중) / ARCHIVED(아카이브) / HIDDEN
 ├ current_revision_id
 └ created_at, updated_at

DebateWikiRevision (debate_wiki_revision)
 ├ revision_id
 ├ wiki_id
 ├ content             본문 (찬/반 논거 포함)
 ├ edited_by
 ├ edit_summary
 └ created_at
```

- `status` 는 `DebateTopic.status` 에 종속된다(OPEN↔ACTIVE, CLOSED↔ARCHIVED). **단일 진실 공급원은 `DebateTopic.status`** 로 두고 위키는 파생값으로 취급하는 편이 상태 불일치를 막는다. (즉 `DebateWiki.status` 를 아예 두지 않고 join 으로 판단하는 선택지도 있음 — 조회 편의 vs 정합성 트레이드오프)
- 이슈 하나에 대한 위키 목록을 뽑으려면 `wiki → topic → chat_room → issue` 3단 join 이 필요하다. 이슈가 채팅방 1:N 이므로 이 경로가 자주 쓰인다 → **`debate_topic.issue_id` 비정규화**를 권장 (3.1).
- 편집 권한은 4.2 매트릭스 참조. 초기엔 ADMIN + 주제 작성자로 좁게 시작하고, 리비전 구조만 미리 잡아두면 나중에 개방이 쉽다.

### 3.4 Event(사건) 신규 도입

```
Event (event)
 ├ event_id
 ├ issue_id            ← 이슈에 부착
 ├ title, summary
 ├ occurred_at         ← 사건 발생 시점 (타임라인 정렬 키)
 ├ status              DRAFT / PUBLISHED / ARCHIVED
 └ created_by, created_at, updated_at
```

- 이슈 상세에서 `occurred_at DESC` 타임라인으로 노출.
- 미디어를 사건 단위로도 묶을 수 있게 매핑 테이블에 `event_id` 를 nullable 로 열어두면 확장 여지가 생긴다.
- 생성 권한: ADMIN.

> **결정 필요**: 사건이 단순 타임라인 항목인가, 아니면 사건에도 채팅방/주제가 붙을 수 있는 토론 단위인가?
> 후자면 `chat_room.event_id` 가 필요해지고 계층이 한 단 깊어진다. **초기엔 타임라인 항목으로 한정**할 것을 권장.

### 3.5 미디어 ↔ 이슈 클러스터링

> **⚠ 이 절은 실측으로 전제가 바뀌었다.** 클러스터링 파이프라인은 **이미 만들어져 있다** (`toronchul_data_v1`, 6.6 참조).
> 아래 "신규 설계" 대신 **기존 파이프라인 산출물과 앱 이슈를 잇는 것**이 실제 과제다. 6.6 을 먼저 읽을 것.

현재 `MediaEntity` 는 `category` 문자열만 있고 이슈와 무관하다. 스케줄러(`domain/youtubeLive/scheduler/*`)가 뉴스·유튜브를 수집해 넣을 뿐이다.

```
IssueMedia (issue_media)
 ├ id
 ├ issue_id
 ├ media_id
 ├ event_id      nullable   ← 사건 단위 묶음용 여지
 ├ score         연관도 (자동 클러스터링용)
 ├ linked_by     AUTO / ADMIN
 └ created_at
 UNIQUE (issue_id, media_id)
```

- 직접 FK(`media.issue_id`) 대신 **매핑 테이블**을 쓰는 이유: 기사 하나가 여러 이슈에 걸칠 수 있고, **자동 클러스터링 결과와 관리자 수동 큐레이션을 한 테이블에 공존**시킬 수 있다.
- `linked_by` 를 두면 관리자가 손댄 연결을 배치가 다시 덮어쓰지 않게 할 수 있다.
- 클러스터링 단계: **1단계** — 관리자 수동 연결 + `category`/키워드 룰 기반 자동 후보 추천(결정론적, 디버깅 가능). **2단계** — 임베딩 유사도 기반 배치로 `score` 채우고 임계치 이상만 노출.

### 3.6 채팅방 진입 시 컨텍스트 로드

`GET /api/v1/room?chatroom-id=` 응답 확장:

```
ChatRoomDetailResponse
 ├ chatRoom   : id, title, content, status
 ├ issue      : id, title, majorCategory, bookmarked(로그인 시)
 ├ topics     : [ { topicId, title, status, agreeCount, disagreeCount, commentCount, myOpinion, wikiId } ]
 │                ├ OPEN 우선 정렬, 아카이브는 별도 조회로 분리 권장
 │                └ **빈 배열 정상** (주제 0개인 채팅방)
 ├ events     : 최신 N건 요약 (선택)
 ├ media      : 해당 이슈로 클러스터링된 미디어 N건  ← 현재의 전역 속보 10건을 대체
 └ presence   : onlineCount
```

- 이 항목의 실질은 `ChatRoomServiceV1.fetch()` 의 **전역** `findTop10BreakingNews()` 를 이슈 클러스터 미디어로 교체하는 것이다.
- 위키 **본문**은 여기 싣지 않는다. 목록엔 제목/상태/찬반만, 본문은 `GET /api/v1/wiki/{wikiId}` 로 분리.

### 3.7 스레드 — 주제에 달린 채팅

토론 주제는 **스레드**다. 별도 화면으로 격리되지 않고, 같은 채팅방 스트림 안에 함께 흐르되 주제로 접어볼 수 있다.

```
ChatEntity + topic_id (nullable, FK → debate_topic)
   NULL  = 일반 발언 (주제 없이 그냥 채팅)
   N     = 주제 N 스레드에 달린 발언
```

**두 가지 조회 모드**

| 모드 | 쿼리 | 용도 |
|---|---|---|
| 전체 스트림 | `WHERE chat_room_id = ?` (topic_id 무관) | 채팅방 기본 화면. 스레드 발언도 인라인으로 함께 보이고, 소속 주제 배지를 단다 |
| 스레드 필터 | `WHERE chat_room_id = ? AND topic_id = ?` | 특정 주제만 골라 보기 |

- 전체 스트림은 **기존 쿼리 그대로**다(`ChatJpaRepository.findByRoomIdAndCursor`). `topic_id` 는 응답에 실어 클라이언트가 배지를 그리게만 하면 된다 → **기존 조회 경로 무회귀**.
- 스레드 필터는 `topic_id` 조건이 붙은 쿼리를 신설. **커서는 모드별로 독립**이어야 한다(전체 스트림 커서를 스레드 뷰에 그대로 쓰면 페이지가 어긋난다).
- **인덱스**: 현재는 방 단위 커서 조회만 있으므로 `chat(chat_room_id, chat_id)` 계열이면 충분했다. 스레드 필터를 위해 **`chat(topic_id, chat_id)` 인덱스 추가**가 필요하다. `chat` 은 서비스에서 가장 빨리 커지는 테이블이므로 인덱스 없이 열면 바로 느려진다.
- 스레드별 댓글 수를 채팅방 상세에 노출하려면 매번 `COUNT` 하지 말고 `debate_topic.comment_count` 를 비정규화 (찬반 카운터와 같은 방식, 3.2).

**날짜별 조회 주의**: `findByRoomIdAndCursorAndDate` 가 `DATE(c.time_stamp) = :date` 로 **컬럼에 함수를 씌워** 인덱스를 못 타는 상태다. 스레드 필터에 같은 패턴을 복사하지 말 것 — 범위 조건(`time_stamp >= :start AND < :end`)으로 쓸 것.

### 3.8 찬반 없이 채팅 참여 (+ opinionType 서버 판정)

기능적으로는 이미 막혀 있지 않으나 데이터 흐름이 어긋나 있다.

- 현재: `ChatMessageRequest.opinionType` 을 클라이언트가 실어 보내고 `ChatEntity.from()` 이 검증 없이 저장 → **위조 가능**.
- v1.3.4: sender 를 서버가 채우듯 **`opinionType` 도 서버가 판정**한다. 찬반이 주제 단위가 되었으므로 기준은 메시지의 `topic_id`:

```
메시지의 topic_id == NULL           → 배지 없음 (NEUTRAL)
메시지의 topic_id == N
   ├ UserTopicOpinion(user, N) 있음 → 그 값 (AGREE / DISAGREE)
   └ 없음                          → 배지 없음 (NEUTRAL)
```

- 즉 **투표하지 않아도 스레드에 발언할 수 있다.** 배지가 안 붙을 뿐이다.

**배지는 발언 시점에 고정(snapshot)한다 — 확정.**

메시지를 저장할 때 그 시점의 투표를 조회해 `chat.opinion_type` 에 함께 기록하고, 이후 조회는 이 컬럼을 그대로 읽는다.

```
14:00  투표 AGREE
14:01  "원전 확대 찬성합니다"   → opinion_type = AGREE 로 저장
14:30  투표를 DISAGREE 로 변경
       → 14:01 메시지는 그대로 [찬성] 배지
```

- **맥락 보존**: 실시간 반영(조회 시 join) 방식이면 입장을 바꾸는 순간 과거 발언 전체의 배지가 소급 변경되어, "찬성합니다"라는 메시지에 [반대] 배지가 붙는 로그가 만들어진다.
- **쿼리 비용**: 채팅 목록은 커서 페이징으로 계속 긁는 가장 뜨거운 경로다. 실시간 반영은 메시지 N건마다 `(user_id, topic_id)` 투표 조회가 붙는다. 고정 방식은 컬럼을 읽으면 끝.
- **변경 폭**: `chat.opinion_type` 컬럼은 이미 존재하므로 **채우는 주체만 클라이언트 → 서버로 바꾸면 된다.** 스키마 변경 없음.
- 트레이드오프: 메시지 배지로는 "그 사람의 현재 입장"을 알 수 없다. 이는 주제 헤더의 찬반 카운트(`agree_count`/`disagree_count`)가 담당한다.

### 3.9 Presence (동시접속자)

**요구**: (a) 전체 접속자(web+mobile 합산), (b) 채팅방별 접속자.

현재 `WebSocketConfig` 에는 CONNECT 인터셉터만 있고 세션 라이프사이클 리스너가 없다.

```
SessionConnectedEvent    → sessionId ↔ userId 등록
SessionSubscribeEvent    → destination(/topic/room{N}) 파싱해 room 카운터 증가
SessionUnsubscribeEvent  → 카운터 감소
SessionDisconnectEvent   → 세션의 모든 구독 정리 + 전체 카운터 감소
```

- **저장소**: 현재 단일 Lightsail 인스턴스 + `SimpleBroker` 구성이므로 **인메모리(`ConcurrentHashMap`)로 시작**. 다중 인스턴스 확장 시 Redis 필요하나 현재 인프라에 Redis 없음.
- **집계 단위**: "총 접속 사용자"라는 표현상 **userId 기준 distinct** 가 맞다(웹+앱 동시 접속 시 세션 2개, 사용자 1명). 익명 연결(dev/local 의 `auth-required:false`)은 userId 가 없으므로 sessionId fallback 필요.
- **노출**: `GET /api/v1/presence` 폴링 + `/topic/room{N}/presence` 로 5~10초 debounce 브로드캐스트.
- web/app 분리가 필요하면 STOMP CONNECT 헤더에 `X-Client-Type: web|app` 을 받아 분리 집계.
- **주의**: 비정상 종료로 DISCONNECT 이벤트가 유실되면 카운터가 샌다(leak). TTL/주기적 reconcile 또는 하트비트 만료를 함께 설계할 것.

### 4.3 인가 적용 방식

`anyRequest().permitAll()` 을 한 번에 걷어내면 회귀 위험이 크다. **인증은 기존 필터에 두고 인가만 Security 로 넘기는 하이브리드**부터:

```java
.authorizeHttpRequests(auth -> auth
    .requestMatchers("/api/v1/admin/**").hasRole("ADMIN")
    .requestMatchers(HttpMethod.POST,   "/api/v1/issue").hasRole("ADMIN")
    .requestMatchers(HttpMethod.POST,   "/api/v1/room").hasRole("ADMIN")   // ← 유저 생성 차단
    .requestMatchers(HttpMethod.PATCH,  "/api/v1/issue/**").hasRole("ADMIN")
    .requestMatchers(HttpMethod.DELETE, "/api/v1/issue/**").hasRole("ADMIN")
    .anyRequest().permitAll()   // ← 최종적으로 authenticated() 로 전환
)
```

- `SecurityPathMatcher` 의 `/api/v1/room` GET-only optional 분기는 그대로 유효 (조회는 비로그인 허용, POST 는 이제 ADMIN).
- `AdminIssueControllerV1` 매핑을 `/api/v1/admin/issue` 로 옮기면 매처가 단순해지나 프론트 계약이 바뀐다 → v1.3.4에서는 **신규 경로 추가 + 기존 경로 유지(deprecated)** 병행 권장.
- 소유권 기반 판정(주제 수정/삭제)은 URL 매처로 불가하므로 **서비스 계층에서 `created_by` 대조**.

## 5. 마이그레이션 전략

기존 `chat_room` 행은 채팅(`chat.chat_room_id`)과 투표(`user_chat_room.chat_room_id`) FK 를 모두 물고 있다.
**`chat_room` 의 PK 는 절대 건드리지 않는다.**

```
현재                                v1.3.4

chat_room #12                       chat_room #12          ← PK 유지, 채팅 컨테이너
 ├ title "원전 확대?"                 ├ title "원전 확대?"    ← 그대로 (방 이름)
 ├ content                          └ issue_id 유지
 └ issue_id
                                    debate_topic #N        ← 신규 생성
user_chat_room                       ├ chat_room_id = 12
 ├ opinion  ─────────────────────┐   ├ title = chat_room.title
 └ interest                      │   ├ status = OPEN (또는 CLOSED, 정책 결정)
                                 │   └ created_by = NULL (시스템 생성)
                                 │
                                 └─> user_topic_opinion    ← opinion 이관
                                      (user_id, topic_id=N, opinion)

                                    user_chat_room         ← interest 만 남김
```

절차:
1. `debate_topic` 에 채팅방별 "첫 주제" 행을 생성 (title/content 복사, `created_by = NULL`)
2. `chat_room_id → topic_id` 매핑을 임시 테이블에 확보
3. `user_chat_room.opinion` 이 NULL 이 아닌 행을 `user_topic_opinion` 으로 복사 (매핑 사용)
4. `debate_wiki` + 초기 리비전(본문 = `chat_room.content`) 생성
5. 검증(건수 대조) 후 `user_chat_room.opinion` 컬럼 drop

**착수 전 필수 확인**: `user_chat_room` 에 `(user_id, chat_room_id)` 중복 행이 있으면 3단계에서 UNIQUE 위반이 난다.
`chat_reaction` 에서 유사 사고 전례 있음 → `docs/chat-reaction-unique-index-issue.md`.
`opinion` 값의 대소문자/오타 혼재 여부도 함께 SELECT 로 확인할 것.

## 6. 배포 · 클라이언트 호환성 전략

### 6.1 제약

- **웹**: 배포 즉시 반영.
- **모바일**: 스토어 심사 3~4일. 심사 통과는 시작일 뿐이고, **실제 사용자 확산은 그보다 훨씬 오래 걸린다.**
- 따라서 **백엔드는 구버전 앱과 신버전 앱을 동시에 지원하는 기간을 반드시 갖는다.**
  (강제 업데이트로 이 기간을 **유한하게 만들 수는 있어도 없앨 수는 없다** — 6.1-1)

### 6.1-1 강제 업데이트를 쓰면 되지 않나 — 대체로 맞다

`GET /api/v1/app/version/check?versionCode=` → `isForceUpdate` (`AppVersionServiceV1`) 가 있으므로,
**구버전 롱테일 문제는 강제 업데이트로 접을 수 있다.** "구버전이 영원히 남는다"를 전제로 한 무기한 이중 지원은 필요 없다.

다만 강제 업데이트가 **없애주지 못하는 것**이 셋 있다.

**(1) 배포 시차 자체는 못 없앤다 — 이게 진짜 제약**

`force_update` 는 **새 빌드가 스토어에 올라간 뒤에야** 켤 수 있다. 그전에 켜면 갈 곳 없는 사용자를 막는 꼴이다.
따라서 순서는 강제되며, 그 사이 구 앱은 계속 살아 있다:

```
T0      백엔드 배포           ← 이 시점부터 구 앱과 공존 시작 (불가피)
T0+3~4일 스토어 심사 통과
T0+n일  force_update ON       ← 여기서야 구버전 차단 가능
그 후    Contract (컬럼 drop, 구 API 차단)
```

즉 **"백엔드는 구 앱과 호환되어야 한다"는 요구는 사라지지 않고, 무기한에서 유한(T0 ~ 스토어 반영)으로 바뀔 뿐이다.**
6.2 Expand/Migrate/Contract 와 6.4 대표 주제 shim 은 **그대로 필요하다.** 달라지는 건 Contract 를 앞당길 수 있다는 것뿐.

**(2) 이 장치는 프로덕션에서 한 번도 검증된 적이 없다**

`app_version` 실측 (2026-07-21):

| version | version_code | force_update |
|---|---|---|
| 0.0.2 | 15 | 0 |
| 1.1.2 | 34 | 0 |
| 1.2.5 | 43 | 0 |
| 1.2.6 | 44 | 0 |

**4개 행 전부 `force_update = 0`.** 한 번도 켜본 적이 없다.
그리고 이 검사는 **서버가 강제하는 게 아니라 앱이 자발적으로 호출하고 존중하는** 구조다 — 백엔드는 클라이언트 버전을 알지 못하므로 구 앱 요청을 거부할 수단이 아예 없다.
즉 **처음 켜보는 장치에 스키마 drop 을 거는 셈**이다. 최소한 v1.3.4 이전에 무해한 버전으로 **한 번 예행 연습**할 것.

**(3) 강제 업데이트는 degrade 가 아니라 차단이다**

플래그를 켜는 순간 구버전 사용자는 기능 일부가 아쉬워지는 게 아니라 **앱을 못 쓴다.**
단계적 출시(phased rollout) 중이거나, OS 버전·저장공간 문제로 업데이트가 불가한 사용자는 그대로 이탈한다.
`version_code 15`(2025-03) 같은 오래된 빌드가 애초에 이 검사를 구현했는지도 확인이 필요하다 — 구현돼 있지 않다면 그 사용자들은 **차단도 안 되고 업데이트도 안 되는** 상태로 구 API 를 계속 때린다.

**결론**: 강제 업데이트를 **주 수단으로 쓰되**, 6.2~6.4 의 호환 설계는 T0~스토어 반영 구간을 위해 유지한다.
아래 텔레메트리는 "드레인을 측정해야 하므로 필수"가 아니라, **강제 업데이트가 실제로 먹혔는지 확인하는 유일한 수단**으로서 값싼 보험이다.

### 6.1-2 클라이언트 버전 텔레메트리 (권장, 필수 아님)

백엔드는 지금 **어떤 앱 버전이 어떤 API 를 때리는지 전혀 모른다.**
`X-Client-Version` / `X-Client-Type` 헤더를 신버전부터 실어 보내게 하고 로깅하면:
- 강제 업데이트가 실제로 먹혔는지 확인 가능
- Contract 시점을 감으로 정하지 않아도 됨
- 3.9 Presence 의 web/app 분리 집계에 같은 헤더를 재사용

필터 한 곳에 추가하는 수준이라 비용이 거의 없다. **차단 로직 없이 로깅만** 넣는 것으로 시작.

### 6.2 원칙 — Expand / Migrate / Contract

스키마와 API 모두 **3단계로 쪼개고, Contract(수축)는 구버전이 빠진 뒤로 미룬다.**

| 단계 | 하는 일 | 배포 시점 |
|---|---|---|
| **Expand** | 새 컬럼·테이블을 **nullable 로 추가**. 기존 것은 아무것도 건드리지 않음 | 즉시 (구/신 클라이언트 모두 무영향) |
| **Migrate** | 데이터 이관 + **이중 기록(dual-write)**. 구 API 는 계속 동작하되 새 구조에도 함께 씀 | 즉시 |
| **Contract** | 구 컬럼 drop, 구 API 차단 | **스토어 반영 + `force_update` ON 이후** (6.1-1) |

API 규칙:
- **응답에 필드를 추가하는 것은 안전**하다 (JSON 파서가 무시).
- **필드 제거·개명·의미 변경은 위험**하다. 필요하면 `/api/v2/...` 로 신규 엔드포인트를 파고 구 엔드포인트를 유지한다 — 이 레포엔 이미 `/api/v2/users/login` 전례가 있다.
- 요청에서 **새 필드는 반드시 optional**. 구 앱은 안 보낸다.

### 6.3-1 클라이언트 실측 (2026-07-22)

형제 레포 3개를 직접 확인했다.

| 레포 | 성격 | 호출하는 이슈/토론 API |
|---|---|---|
| `DebateSeason_Frontend_V1` | Flutter (모바일) | `GET /api/v1/room`, `POST /api/v1/room/vote`, `GET /api/v1/issue` |
| `toronchul_web_v1` | Next.js (웹) | `GET /api/v1/room`, `POST /api/v1/room/vote`, `GET /api/v1/issue` |
| `debate-season-admin` | **비어 있음** (파일 없음) | — |

**결론**
- **`POST /api/v1/room`(채팅방 생성) / `POST /api/v1/issue`(이슈 생성) 을 호출하는 클라이언트가 하나도 없다.**
  → §7 1번 인가 매처 적용은 **403 회귀 위험이 없다.** 즉시 적용 가능.
- **레거시 STOMP `chat.sendMessage` / `chat.join` / `/topic/public` 사용처도 0건.** → v1.3.1 에서 제거 완료.
- 구 앱이 실제로 쓰는 쓰기 API 는 **`POST /api/v1/room/vote` 하나뿐**이다. → 6.4 대표 주제 shim 이 커버해야 할 범위가 이것 하나로 확정.

> **새로 드러난 문제**: 어드민 레포가 비어 있다. 즉 **이슈/채팅방을 만들 운영 UI 가 존재하지 않는다.**
> 지금까지 Swagger 나 DB 직접 삽입으로 운영해 온 것으로 보인다.
> 채팅방 생성을 ADMIN 전용으로 잠그면 **운영 수단을 함께 마련해야 한다** (최소한 ADMIN 계정 + Swagger 접근).

### 6.4 구 앱 투표를 어디로 보낼 것인가 (가장 까다로운 지점)

구 앱은 "채팅방에 찬성/반대"라는 개념만 안다. 신 구조엔 그런 게 없다.

**대표 주제(default topic) shim** 을 둔다:
- 마이그레이션에서 채팅방마다 만드는 "첫 주제"(§5)를 그 방의 **대표 주제**로 지정한다 (`chat_room.default_topic_id`).
- 구 API `POST /api/v1/room/vote` 는 그 방의 **대표 주제에 투표**하는 것으로 라우팅한다.
- 구 API `GET /api/v1/room` 의 `agree`/`disagree`/`opinion` 도 **대표 주제 기준**으로 채워 응답한다.
- 이렇게 하면 구 앱은 아무것도 모른 채 계속 동작하고, 신 앱은 주제별 투표를 쓴다. **`user_chat_room.opinion` 을 dual-write 할 필요조차 없다** — 단일 진실은 `user_topic_opinion` 이 된다.

> **결정 필요**: 대표 주제가 CLOSED 되면 구 앱 투표는 어디로 가는가? (상세는 §0.3 결정 3)
> (a) 가장 최근 OPEN 주제로 자동 이동, (b) 투표 거부(구 앱에 에러), (c) 대표 주제는 자동 종료 대상에서 제외.
> **(c) 권장.** (a) 는 유저가 의도하지 않은 주제로 표가 들어가 데이터가 오염된다.

### 6.5 실측 — 마이그레이션 리스크는 예상보다 낮다

| 테이블 | 행 수 | 비고 |
|---|---|---|
| `chat` | **386** | AUTO_INCREMENT=520. **`ALTER TABLE` 무중단 고민 불필요** |
| `user_chat_room` | 453 | **`(user_id, chat_room_id)` distinct 453 = 중복 0건 → UNIQUE 인덱스 안전** |
| `chat_room` | 26 | 대표 주제도 26건만 생성하면 됨 |
| `issue` | 17 | |
| `users` | 231 | role 백필 대상 소규모 |
| `user_issue` | **0** | 북마크 기능 **사용 이력 없음** |
| `media` | 19,681 (6.5MB) | 최대 테이블 |

- MariaDB **10.6.22**. `chat` 이 386행이므로 `ADD COLUMN` / `ADD INDEX` 모두 순간이다. §7 3번 항목의 "대용량 ALTER" 우려는 **해소**.
- **`chat` 에 커서 페이징용 복합 인덱스가 없다** (`chat_room_id`, `user_id` 단일 인덱스만). 지금은 데이터가 작아 문제가 안 되지만, `topic_id` 인덱스를 넣을 때 `(chat_room_id, chat_id)` 도 같이 정리할 것.
- `user_issue` 가 0행이므로 북마크 관련 마이그레이션은 사실상 무비용.

### 6.6 이미 존재하는 클러스터링 자산 (코드에 없음)

prod DB 에 **JPA 엔티티가 없는 테이블**이 있다. 3.5 미디어 클러스터링을 백지에서 설계할 필요가 없을 수 있다.

| 테이블 | 행 수 | 내용 |
|---|---|---|
| `keyword` | 32,390 | `id, name` — 키워드 사전 |
| `media_keyword` | 32,354 | `media_id × keyword_id` — **미디어 키워드 추출 결과** |
| `create_issue` | 21 | `issue_id(UUID), issue_name, **issue_vector(mediumtext)**, status, article_count` — **임베딩 기반 이슈 클러스터링 산출물** |

**파이프라인을 찾았다: `toronchul_data_v1` (형제 레포, Python)**

```
main.py                  CLI: python main.py [YYYY-MM-DD]
config/db.py             SSH 터널 + PyMySQL → Lightsail MariaDB
create_issue/
 ├ embedder.py           jhgan/ko-sroberta-multitask 임베딩
 ├ clusterer.py          2단계 군집화 (HDBSCAN 엄격 → Agglomerative cosine 0.4)
 ├ issue_matcher.py      기존 이슈와 코사인 유사도 + 시간 감쇠 매칭
 │                        ≥0.75 confirm(EMA α=0.3 로 벡터 갱신) / ≥0.65 pending / 그 외 new
 ├ issue_namer.py        LLM 이슈명 생성 — **미사용** (지금은 "클러스터_001" 순번)
 └ db_ops.py             create_issue CRUD + media.create_issue_id 링크
```

- 처리 흐름: `media` 에서 당일 `type='news'` 타이틀 조회 → 임베딩 → 군집화 → 기존 이슈 매칭 → `create_issue` INSERT/UPDATE + **`media.create_issue_id` 갱신**.
- 즉 3.5 에서 "신규 설계"라 적었던 것 — 임베딩 클러스터링, 연관도 판정, 미디어↔이슈 연결 — 이 **전부 이미 구현돼 있다.**

**따라서 v1.3.x 의 실제 과제는 클러스터링이 아니라 "이어붙이기"다.**

```
media ──create_issue_id──> create_issue        ← 파이프라인이 이미 채움 (자동 클러스터)
                              ?
                              ↕  ← 이 연결이 없다
                              ?
                           issue               ← 앱이 노출하는 큐레이션 이슈 (bigint PK, 17건)
```

- `create_issue.issue_id` 는 **UUID(varchar 36)**, 앱 `issue.issue_id` 는 **bigint**. 완전히 별개 테이블이다.
- 연결 방식 후보: (a) `issue.create_issue_id` 컬럼 추가 (1:1), (b) `issue_cluster(issue_id, create_issue_id)` 매핑 (N:M).
  자동 클러스터 여러 개가 한 큐레이션 이슈에 묶일 수 있으므로 **(b) 권장.**
- 연결되면 "이슈에 클러스터링된 미디어" = `issue → create_issue → media` 2-hop 조회로 끝난다.
  **3.5 에서 제안한 `issue_media` 매핑 테이블은 불필요하다** — 이미 `media.create_issue_id` 가 그 역할을 한다.

**현재 상태 — 멈춰 있음**

| 지표 | 값 |
|---|---|
| `create_issue` | 21건, 전부 `2026-03-03` 하루치 |
| `media.create_issue_id` 연결된 행 | **19,729건 중 59건** (0.3%) |
| 이슈명 | `클러스터_001` 형식 (LLM 네이밍 미적용) |
| 실행 방식 | 수동 CLI. prod 크론·systemd timer 에 등록돼 있지 않음 |

즉 **하루치 시범 실행 후 멈춘 상태**다. 자산으로서 가치는 크지만 그대로 갖다 쓸 수는 없다.

> **결정 필요**: v1.3.7 에서 (a) 파이프라인을 정기 실행으로 되살리고 앱 이슈와 연결할 것인가,
> (b) 아니면 v1.3.4 에서는 **관리자 수동 큐레이션만** 넣고 자동화는 뒤로 미룰 것인가?

## 8. 개발 착수 전 체크리스트

### 8.1 스키마 변경 운영 방식 — 이 레포의 기존 규약

- prod 는 `ddl-auto: none`. **스키마는 수동 SQL 로 적용하고 `scripts/migrations/` 에 이력을 남긴다.**
  (선례: `2026-07-20__profile_age_range_add_undefined.sql` — 배경/영향/적용여부를 주석으로 남기는 형식)
- **dev·local 은 `ddl-auto: update`** 라 엔티티만 추가하면 자동 생성된다.
  → **여기가 드리프트 위험 지점이다.** 로컬은 Hibernate 가 만든 스키마, prod 는 손으로 쓴 SQL 이므로
  컬럼 타입·인덱스명·FK 옵션이 조용히 어긋날 수 있다.

**권장**: 손으로 DDL 을 쓰지 말고, 엔티티 작성 후 **Hibernate 스키마 생성 결과를 뽑아 그것을 마이그레이션 파일의 출발점으로 삼는다.**
(`spring.jpa.properties.jakarta.persistence.schema-generation.scripts.action=create` 로 로컬에서 1회 추출)
그 뒤 인덱스·FK 를 손보고 `scripts/migrations/` 에 커밋.

> **결정 필요**: 신규 테이블이 5개(+ALTER 다수)라 수동 관리 부담이 커진다. **Flyway 도입 시점인가?**
> 권장: **v1.3.x 동안은 도입하지 않는다.** 기존 규약이 명확하고, 마이그레이션이 단계별로 쪼개져 있어 수동으로 감당 가능하다.
> 도입한다면 v1.3.x 완료 후 baseline 을 떠서 별도로 하는 편이 안전하다.

### 8.2 배포 단위와 롤백

- 7장의 각 단계는 **독립 배포 가능하도록** 설계돼 있다. 한 번에 다 올리지 않는다.
- **§7 2번(데이터 이관) 직전에 수동 백업을 뜬다.** 일일 백업은 KST 04:00(`scripts/db-backup.sh`) 이므로 그것만 믿지 않는다.
- 이관 스크립트는 **재실행 가능(idempotent)** 하게 쓴다. 중간 실패 시 처음부터 다시 돌릴 수 있어야 한다.
- Contract 단계(컬럼 drop)는 **되돌릴 수 없다.** 6.2 순서를 지킨다.
