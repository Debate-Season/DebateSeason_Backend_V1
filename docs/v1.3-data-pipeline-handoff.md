# v1.3.x — 데이터 레포(`toronchul_data_v1`) 작업 요청서

- 작성일: 2026-07-22
- 대상 레포: `toronchul_data_v1` (Python, 뉴스 임베딩·군집화 파이프라인)
- 관련 문서: `docs/v1.3.3-issue-debate-structure-prd.md` §3.5, §6.6, §6.7
- 백엔드 담당 배포: **v1.3.6** (`issue` ↔ 클러스터 연결) — v1.3.3 범위 밖

---

## 1. 배경 — 왜 이 요청이 생겼나

v1.3에서 **"이슈 상세에 그 이슈로 클러스터링된 미디어를 노출"** 하는 요구가 생겼다.

백엔드 코드만 보면 이건 신규 개발로 보였다. `MediaEntity` 에 `issue_id` 가 없고,
`MediaJpaRepository` 는 `type`/`created_at` 기준 전역 조회만 한다. 채팅방 상세도
**전역** 속보 10건(`mediaManager.findTop10BreakingNews()`)을 보여줄 뿐 이슈와 무관하다.

그런데 prod DB 를 확인해보니 **클러스터링은 이미 구현돼 있었다.**
`toronchul_data_v1` 이 그것이고, `create_issue` 테이블과 `media.create_issue_id` 컬럼을
이미 채우고 있다. 즉 v1.3의 실제 과제는 클러스터링 개발이 아니라 **이어붙이기**다.

### 2.2 원본 데이터 공급이 끊겨 있음 (선행 문제)

파이프라인 입력은 `media` 테이블의 당일 `type='news'` 타이틀이다. 그런데:

| type | 건수 | 최종 수집 시각 |
|---|---|---|
| news | 12,832 | **2026-06-30** (약 3주 전) |
| COMMUNITY | 6,730 | **2026-06-30** |
| youtube | 167 | **2025-07-27** (약 1년 전) |

`youtube` 중단은 백엔드 CLAUDE.md 에 기록된 YouTube API 할당량 소진 이슈와 일치한다.
그러나 **news/COMMUNITY 가 6/30 에 동시에 끊긴 것은 별개 원인**일 가능성이 높다
(백엔드 스케줄러 예외, 소스 사이트 구조 변경 등). 이건 백엔드 쪽에서 조사한다.

> **파이프라인을 되살려도 신규 입력이 없으면 의미가 없다.** 아래 작업의 선행 조건이다.

### 2.3 두 개의 "이슈"가 서로 모른다 — 핵심 문제

```
media ──create_issue_id──> create_issue     ← 파이프라인이 채움
  (19,729행)      (59행 연결)     (21행, UUID varchar(36))
                                      ?
                                      ↕   ← 이 연결이 없다
                                      ?
                                   issue    ← 앱이 사용자에게 노출하는 큐레이션 이슈
                                            (17행, bigint PK, 관리자가 수동 생성)
```

| | `create_issue` | `issue` |
|---|---|---|
| PK | `issue_id` **UUID varchar(36)** | `issue_id` **bigint** |
| 생성 주체 | 파이프라인 (자동) | 관리자 (수동) |
| 행 수 | 21 | 17 |
| 이름 | `클러스터_001` | 사람이 쓴 제목 |
| 사용처 | **없음** (앱이 안 읽음) | 앱 전체 |

**완전히 별개 테이블이고, 이름만 비슷하다.** 이게 연결되지 않으면
"이슈에 클러스터링된 미디어"는 구현할 수 없다.

## 3. 요청 사항

### 3.1 [필수] `create_issue` ↔ `issue` 연결 규약 합의

백엔드는 `issue_cluster(issue_id BIGINT, create_issue_id VARCHAR(36))` **매핑 테이블**을
만들 계획이다 (1:1 컬럼이 아니라 N:M 매핑 — 자동 클러스터 여러 개가 하나의 큐레이션 이슈에
묶일 수 있으므로).

연결되면 이슈별 미디어 조회는 아래 2-hop 으로 끝난다:

```sql
SELECT m.* FROM media m
JOIN issue_cluster ic ON m.create_issue_id = ic.create_issue_id
WHERE ic.issue_id = ?
ORDER BY m.created_at DESC
```

**데이터 레포에 요청하는 것**: 이 매핑을 **누가 채우는가**를 정해야 한다.

| 방식 | 설명 | 판단 |
|---|---|---|
| (a) 관리자 수동 | 백엔드 어드민에서 "이 클러스터를 이 이슈에 연결" | **v1.3 권장 출발점** |
| (b) 파이프라인 자동 제안 | 파이프라인이 `issue` 제목과 클러스터 centroid 를 비교해 후보 제시 | v1.4 이후 |
| (c) 완전 자동 | 파이프라인이 `issue` 행까지 생성 | 이름 품질 문제로 시기상조 |

→ **(a) 로 시작하고 (b) 를 준비**하는 것을 제안. 파이프라인은 지금처럼
`create_issue` / `media.create_issue_id` 만 채우면 되고, 매핑은 백엔드가 관리한다.

**단, (b) 를 하려면 파이프라인이 `issue` 테이블을 읽어야 한다.** 그때 필요한 것:
- `issue.title` 임베딩 → 각 `create_issue.issue_vector` 와 코사인 유사도
- 임계치 이상인 조합을 `issue_cluster` 에 `linked_by='AUTO'`, `score` 와 함께 제안
- 관리자가 확정/거부 → `linked_by='ADMIN'` 으로 승격, 이후 배치가 덮어쓰지 않음

### 3.2 [필수] 이슈명 생성(`issue_namer.py`) 활성화

현재 `issue_name` 이 `클러스터_001` 형식이다. **이 상태로는 사용자에게 노출할 수 없다.**
`instruction.md` 에 "추후 Claude API 연동 시 `issue_namer.py` 활성화 예정"이라고 적혀 있다.

- 3.1의 (b) 자동 제안을 하려면 이름 품질이 먼저 올라와야 한다
  (제목 유사도로 매칭할 것이므로).
- (a) 수동 연결만 한다면 관리자가 클러스터를 식별할 수 있는 수준이면 충분하다.
  최소한 **대표 기사 제목 몇 개를 이름 대신 보여줄 수 있게** 결과에 포함해달라.

### 3.3 [필수] 정기 실행 체계

현재 수동 CLI 라 크론/systemd timer 어디에도 없다. 되살릴 때:

- 실행 주기: **일 1회** (파이프라인이 "당일 뉴스 1일치" 전제로 설계돼 있음)
- 실행 위치: prod 서버에 올릴지, 로컬/별도 환경에서 SSH 터널로 돌릴지 결정 필요
  (현재 `config/db.py` 가 **SSH 터널 + PyMySQL** 구조 → 외부 실행 전제로 보임)
- **멱등성**: 같은 날짜로 재실행해도 중복 이슈가 생기지 않아야 한다. 중간 실패 시
  재실행이 필요하므로.
- **실패 알림**: 백엔드에 `scripts/error-monitor.sh` (10분 주기 Discord 알림) 선례가 있다.
  같은 방식으로 붙이면 이번처럼 4개월간 조용히 멈춰 있는 상황을 막을 수 있다.
- **공백 기간 처리**: 2026-03-03 이후 데이터가 없다. 재개 시 그 사이 `media` 를
  소급 처리할지(백필), 재개 시점부터만 할지 결정 필요.

### 3.4 [주의] 스키마 자동 변경

`db_ops.py:6` 이 **`media.create_issue_id` 컬럼이 없으면 파이프라인이 자동으로 추가**한다.

백엔드는 prod 를 `ddl-auto: none` 으로 두고 스키마 변경을 `scripts/migrations/` 에
수동 SQL 로 관리한다. **두 시스템이 같은 테이블 스키마를 서로 모르게 건드리면 위험하다.**

요청:
- 파이프라인의 자동 `ALTER TABLE` 을 **제거하거나 최소한 로그를 남기도록** 해달라.
- 앞으로 `media` 스키마 변경이 필요하면 백엔드 `scripts/migrations/` 에 SQL 을 추가하는
  방식으로 통일한다.
