- 선행: v1.3.1 배포 완료 (2026-07-22)

## 1. 계기

v1.3.1 배포 검증용 스모크 테스트에서 존재하지 않는 채팅방(`chatroom-id=1`)을 조회했다.
정상적인 404 응답이 나갔는데 **Discord 에 장애 알림이 발송됐다.**

```
🚨 toronchul 에러 감지 — 1건
▸ 1건 · c.d.c.exception.GlobalExceptionHandler | CustomException occurred. ErrorCode: <n>, Message: null
```

서비스에는 아무 문제가 없었다. 알림 파이프라인이 **클라이언트 오류를 서버 장애로 보고**한 것이다.

## 2. 문제

### 2.1 모든 CustomException 이 ERROR 로 기록된다

`GlobalExceptionHandler.handleCustomException` 이 상태 코드와 무관하게 `log.error` 를 호출했다.
`error-monitor.sh` 는 `' ERROR | SEVERE '` 패턴을 잡아 Discord 로 보내므로,
**클라이언트가 없는 리소스를 요청하기만 해도 알림이 나간다.**

같은 문제가 세 곳 더 있었다. 전부 항상 4xx 인 경우다.

| 핸들러 | 상태 | 기존 레벨 |
|---|---|---|
| `handleCustomException` | 4xx/5xx 혼재 | ERROR |
| `handle(HttpRequestMethodNotSupported)` | 405 | ERROR |
| `handleMethodArgumentNotValid` | 400 (검증 실패) | ERROR |
| `handleMethodArgumentTypeMismatch` | 400 (파라미터 형식) | ERROR |

**입력값 검증에 실패하기만 해도 Discord 가 울리는 상태였다.**

### 2.2 알림 본문으로 원인을 알 수 없다

`ErrorCode: <n>, Message: null`

- `Message: null` — `CustomException(ErrorCode)` 로 던지면 `getMessage()` 가 null 이다.
  대부분의 호출부가 이 생성자를 쓴다.
- `ErrorCode: 404` — 숫자 코드만 찍혀 어떤 에러인지 알 수 없다(`NOT_FOUND_CHATROOM` 인지 `NOT_FOUND_ISSUE` 인지).
- 요청 URI 가 없어 어느 엔드포인트인지도 알 수 없다.

### 2.3 왜 지금 고치는가 — v1.3.3 에서 구조적으로 악화된다

지금은 `chat_room` 26건에 트래픽도 적어 드물게 터진다. 그러나 v1.3.3 에서:

- 소프트 삭제(`DELETED`/`HIDDEN`)와 `status` 조회 필터를 켜면 **"없는 리소스" 404 가 정상 흐름이 된다.**
- 아카이브된 주제·숨겨진 채팅방을 구 앱이 계속 요청한다.

즉 **알림 소음이 구조적으로 늘어난다.** 소음이 늘면 진짜 장애 알림을 놓친다.
v1.3.3 은 덩치가 커서 배포까지 시간이 걸리므로, 이것만 떼어 먼저 올린다.

## 3. 변경 내용

### 3.1 로그 레벨을 상태 코드로 가른다

```
4xx (클라이언트 원인) → log.warn   + 스택트레이스 없음  → 알림 대상 아님
5xx (서버 결함)       → log.error  + 스택트레이스 있음  → 알림 대상
```

- 4xx 에 스택트레이스를 남기지 않는 이유: 원인이 요청 자체라 추적 가치가 낮고 로그만 커진다.
- 항상 4xx 인 나머지 세 핸들러도 `warn` 으로 내렸다.
- **`CustomException` 이 아닌 예외는 그대로다.** `@ExceptionHandler(Exception.class)` 가 없어
  예상치 못한 예외는 컨테이너가 ERROR 로 남긴다 → 알림 계속 발송. 의도한 동작이다.

### 3.2 로그 본문을 판단 가능하게

```
CustomException. code=NOT_FOUND_CHATROOM status=404 uri=/prod/api/v1/room message=주어진 chatroomId로 ...
```

- 숫자 코드 대신 **enum 이름**(`errorCode.name()`)
- **요청 URI** 추가 (`HttpServletRequest` 주입)
- `getMessage()` 가 null 이면 **`ErrorCode` 의 기본 메시지로 fallback**

### 3.3 테스트

`GlobalExceptionHandlerLogLevelTest` — Logback `ListAppender` 로 실제 기록된 레벨을 검증한다.
**로그 레벨이 곧 알림 정책이므로 테스트로 고정한다.**

| 케이스 | 기대 |
|---|---|
| `NOT_FOUND_CHATROOM` (404) | WARN — 이번 오탐을 만든 바로 그 케이스 |
| `INTERNAL_SERVER_ERROR` (500) | ERROR |
| 메시지 없이 던진 경우 | 본문에 enum 이름·URI 포함, `null` 미포함 |
| 지원하지 않는 HTTP 메서드 (405) | WARN |

> **함정**: `application-test.yml` 이 `logging.level.root: error` 로 설정돼 있어,
> Spring 컨텍스트를 띄우는 테스트가 먼저 실행되면 Logback 전역 레벨이 ERROR 가 되어
> **WARN 이벤트가 통째로 잘린다.** 즉 테스트 실행 순서에 따라 결과가 달라진다.
> 실제로 로컬 전체 실행은 통과했는데 CI 에서 3건이 실패했다(로컬은 `common.exception` 이
> 먼저 정렬돼 Spring 테스트보다 앞서 돌았다).
> → 테스트가 `@BeforeEach` 에서 **자기 로거 레벨을 직접 고정**하고 `@AfterEach` 에서 복원한다.
> 전역 로깅 설정에 의존하지 않게 만드는 것이 옳다.

## 4. 호환성

**API 응답은 전혀 바뀌지 않는다.** 상태 코드도 `ErrorResponse` 본문도 그대로다.
로그 레벨과 로그 문구만 바뀐다. 클라이언트 영향 없음, 마이그레이션 없음.

## 5. 배포

`main` push → PROD_CI/CD 자동 배포. DB 변경 없음.

### 검증

1. 배포 후 존재하지 않는 리소스를 조회한다.
   ```
   curl -s "https://api.toronchul.app/prod/api/v1/room?chatroom-id=1"
   ```
   → 응답은 그대로 404 `NOT_FOUND_CHATROOM`
2. 로그에 ERROR 가 아닌 WARN 으로 남는지 확인한다.
   ```
   sudo journalctl -u toronchul.service -S '5 min ago' --no-pager | grep -E 'WARN|ERROR' | grep CustomException
   ```
3. 다음 `error-monitor.sh` 주기(10분)에 **Discord 알림이 오지 않아야 한다.**

### 롤백

애플리케이션만 되돌리면 된다. 상태 변경 없음.

## 6. 남겨둔 것

- `error-monitor.sh` 의 `BENIGN_PATTERNS` 는 손대지 않았다.
  로그 문구 매칭은 취약하고, 서버가 잘못 던진 404 까지 함께 묻힐 수 있다.
  **원인을 로그 레벨에서 바로잡는 편이 옳다.**
- `signature does not match` 를 일부러 거르지 않는 기존 정책도 그대로다.
  (위조 시도이거나 JWT 시크릿 변경 흔적일 수 있어 눈에 띄는 게 낫다)
