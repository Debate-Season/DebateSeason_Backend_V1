# v1.3.1 PRD — 권한 기반 정비 및 보안 결함 수정

- 작성일: 2026-07-22
- 상태: **운영 배포 완료 (2026-07-22)**
- 후속: `docs/v1.3.2-issue-debate-structure-prd.md` (이슈/토론 구조 재편)

---

## 1. 이 배포가 하는 일

v1.3.2의 구조 개편은 "이슈·채팅방 생성을 ADMIN 전용으로 잠근다"를 전제로 한다.
그런데 **이 프로젝트에는 역할(Role) 개념이 코드에 존재하지 않았다.** 잠글 대상은 있는데
잠글 수단이 없었다.

v1.3.1은 구조를 건드리지 않고 **그 수단만 먼저 만든다.** 덧붙여 조사 과정에서 발견된
보안 결함 3건을 함께 고친다.

**이 배포는 사용자에게 보이는 동작을 바꾸지 않는다.** 응답 스펙도 그대로다.
컬럼과 클레임을 추가하는 Expand 단계이며, 인가는 아직 걸지 않는다.

---

## 2. 배경 — 무엇이 문제였나

### 2.1 역할이 없다

| 계층 | 상태 |
|---|---|
| Spring Security | `anyRequest().permitAll()` — URL 인가 전무 |
| 실질 게이트 | `JwtAuthenticationFilter` 의 PUBLIC/OPTIONAL 분기 = "로그인 했는가" 하나뿐 |
| 역할 | `CustomUserDetails` 가 **전원 `ROLE_USER` 하드코딩** |
| `users` 테이블 | role 컬럼 없음 |
| JWT | role 클레임 없음 |
| ADMIN | 코드 전체에서 **주석과 Swagger 설명에만** 존재 (`AdminIssueControllerV1.java:15`) |

즉 `POST /api/v1/issue` 는 "ADMIN 전용"이라고 주석에 적혀 있지만 실제로는
토큰만 있으면 누구나 호출할 수 있었다.

### 2.2 작성자를 기록하지 않는다

`ChatRoomControllerV1.save()` 는 principal 을 파라미터로 받지도 않았고,
`issue` / `chat_room` 어디에도 `created_by` 가 없었다.
**"누가 만들었는가"를 모르므로 수정·삭제 권한을 판정할 근거 자체가 없었고**,
실제로 수정/삭제 API 도 존재하지 않았다(생성+조회만).

### 2.3 보안 결함 3건

1. **신고자 위조** — `ReportController.reportChatMessage` 의 `Long userId` 가 어노테이션 없이
   선언되어 Spring 이 **요청 파라미터로 바인딩**했다. `?userId=999` 를 붙이면 남의 이름으로
   신고할 수 있었다. 정작 호출하던 `authService.getCurrentUserId()` 반환값은 버려졌다.
2. **`@AuthenticationPrincipal` 오용** — `ChatControllerV1` 이 `@AuthenticationPrincipal Long userId`
   로 받았는데 principal 실제 타입은 `CustomUserDetails` 라 **항상 null** 이었다.
   이를 컨트롤러가 `JwtUtil` 로 토큰을 직접 재파싱하는 fallback 으로 덮고 있었다.
3. **레거시 STOMP** — `chat.sendMessage` → `/topic/public` 핸들러가 인증도 저장도 없이
   클라이언트 페이로드를 그대로 브로드캐스트했다(발신자 스푸핑 경로). `chat.join` 도 동일.

### 2.4 강제 업데이트가 먹혔는지 확인할 수단이 없다

`app_version.force_update` 는 존재하나 **4개 행 전부 0 — 한 번도 켠 적이 없다.**
게다가 백엔드는 클라이언트 버전을 전혀 모르므로, 구 API 차단 시점을 판단할 근거가 없다.

## 3. 변경 내용

### 3.1 Role 모델

```
UserRole (USER, ADMIN)
 ├ getAuthority()   → "ROLE_" + name()   (Spring Security 의 hasRole 전제)
 └ fromNullable()   → 클레임 부재/미지 값을 USER 로 격하
```

- `UserEntity` / `User` / `UserMappingData` 에 role 전파
- `JwtUtil`: **access token 에만** role 클레임 추가, `getRole()` 제공
- `CustomUserDetails`: 하드코딩 제거 → 실제 role 기반 authority

**refresh token 에는 role 을 싣지 않는다.** 재발급에만 쓰이고 인가에 쓰이지 않기 때문이다.
대신 `AuthServiceV1` 이 **재발급 시점에 DB 에서 role 을 조회**한다.
덕분에 권한 변경이 다음 재발급에 반영된다(access token 수명이 짧아 지연이 작다).

### 3.2 작성자·상태 컬럼

```
issue      + created_by(NULL 가능), status(IssueStatus),    updated_at
chat_room  + created_by(NULL 가능), status(ChatRoomStatus), updated_at

IssueStatus    : DRAFT, PUBLISHED, ARCHIVED
ChatRoomStatus : OPEN, CLOSED, HIDDEN, DELETED
```

- 채팅방·이슈 생성 시 principal 에서 작성자를 기록
- **`@PrePersist` 로 status 기본값을 보장한다.** `@Builder` 가 필드 초기화식을 무시해서
  생성 경로마다 status 를 빠뜨릴 수 있고, 실제로 기존 테스트 픽스처가 NOT NULL 위반으로
  깨졌다. 호출자에게 맡기지 않고 엔티티가 스스로 보장하게 했다.

**조회 필터는 걸지 않는다.** 컬럼만 추가하고 목록/조회 쿼리는 그대로 둔다.
지금 필터를 걸면 기본값 실수 하나로 프론트에서 목록이 통째로 비어 보일 수 있다.
노출 규칙은 v1.3.2 에서 위키 상태와 함께 켠다.

### 3.3 보안 수정

| 대상 | 조치 |
|---|---|
| `ReportController` | `@AuthenticationPrincipal CustomUserDetails` 로 교체. **회귀 테스트 추가** (요청 파라미터를 실어도 인증된 userId 만 쓰이는지 검증) |
| `ChatControllerV1` | principal 타입 수정, 토큰 재파싱 fallback 및 `JwtUtil` 의존 제거 |
| `WebSocketControllerV1` | 레거시 핸들러 2개 제거. 클라이언트 3개 레포에서 **사용처 0건** 확인 후 삭제 |

### 3.4 클라이언트 버전 로깅

`ClientInfoFilter` 신규 — `X-Client-Type` / `X-Client-Version` 을 MDC 에 기록.
**차단하지 않고 로깅만** 한다. 헤더가 없으면 `unknown`.

> 구현 메모: `addFilterBefore` 의 앵커는 Spring Security 가 아는 필터여야 한다.
> 커스텀 필터를 앵커로 쓰면 컨텍스트가 뜨지 않는다. 같은 앵커에 등록 순서로 배치했다.

## 4. 호환성

**구/신 클라이언트 모두 무영향이다.**

| 변경 | 구버전 앱 영향 |
|---|---|
| role 클레임 추가 | 기존 토큰엔 클레임 없음 → `USER` fallback. **재로그인 불필요**, 만료 주기 내 자연 수렴 |
| `created_by`/`status`/`updated_at` 컬럼 | 응답 스펙 변화 없음 |
| 신고자 principal 수정 | 구 앱은 `userId` 파라미터를 보내지 않으므로 무영향. 보내더라도 무시됨 |
| 레거시 STOMP 제거 | 클라이언트 실측 결과 **호출처 0건** |
| `ClientInfoFilter` | 헤더 없으면 `unknown` 으로 기록만 |

클라이언트 실측 (2026-07-22):

| 레포 | 호출하는 이슈/토론 API |
|---|---|
| `DebateSeason_Frontend_V1` (Flutter) | `GET /room`, `POST /room/vote`, `GET /issue` |
| `toronchul_web_v1` (Next.js) | `GET /room`, `POST /room/vote`, `GET /issue` |
| `debate-season-admin` | **비어 있음 (파일 없음)** |

## 5. 배포 절차

### 5.1 마이그레이션 (수동 적용)

이 프로젝트는 Flyway 를 쓰지 않고 prod 가 `ddl-auto: none` 이므로 수동 적용한다.

```
scripts/migrations/2026-07-22__users_add_role.sql
scripts/migrations/2026-07-22__issue_chatroom_add_ownership_and_status.sql
```

**순서**: DB 마이그레이션 → 애플리케이션 배포.
(컬럼 추가는 구버전 애플리케이션에 무해하므로 먼저 적용해도 안전하다)

대상 규모: `users` 231행, `issue` 17행, `chat_room` 26행 — 전부 소규모라 ALTER 는 순간이다.

**적용 완료 (2026-07-22)** — 백업(`debate_season-20260721-155230.sql.gz`) 후 순서대로 적용.

### 5.3 ADMIN 계정 승격

**v1.3.2 의 인가 매처를 켜기 전에 반드시 선행**해야 한다. 안 하면 이슈·채팅방을
만들 수 있는 계정이 하나도 없게 된다.

```sql
UPDATE users SET role = 'ADMIN' WHERE user_id IN (...);   -- 대상 user_id 사전 확인 필요
```

> 이번 배포에서는 인가를 걸지 않으므로 승격이 급하지는 않다. 다만 v1.3.2 전에는 필수다.

### 5.5 롤백

컬럼 추가와 클레임 추가뿐이라 **애플리케이션만 되돌리면 된다.**
컬럼이 남아 있어도 구버전 애플리케이션은 무시한다. DROP 은 불필요하다.

## 6. 남은 과제 (v1.3.2 로 이월)

- **인가 매처 적용** — 이슈·채팅방 생성 ADMIN 전용화.
  클라이언트 호출처가 0건이라 403 회귀 위험은 없으나, **어드민 UI 가 없어
  운영 수단을 함께 마련해야 한다**(최소한 ADMIN 계정 + Swagger 접근).
  → 이 때문에 v1.3.1 에서 잠그지 않고 v1.3.2 로 미뤘다. 잠금과 운영 수단은 같이 가야 한다.
- **`status` 조회 필터** — 노출 규칙은 위키 상태와 함께 켠다.
- **수정/삭제 API** — `created_by` 가 생겼으므로 이제 소유권 판정이 가능하다.
